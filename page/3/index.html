[译]java多线程与同步机制系列(3)－－多线程的代价

By 朱家骅

发表于 2015-04-23

原文链接(http://tutorials.jenkov.com/java-concurrency/costs.html)


多线程相对于单线程有优势，但是也有劣势，使用多线程使得我们不得不付出一些代价，因此，不要因为自己了解多线程，就在开发过程中不假思索的使用多线程。应该在开发之前认真思考，是否在该项目中使用多线程的好处会大于坏处。当你对于是否使用多线程感到困惑时，尝试着测试程序的运行效率和响应性来作出确定，而不要仅仅靠猜测！

更复杂的设计

在多线程程序中，某些部分的设计确实比单线程设计简单，但是其他部分的设计却比较复杂，比如说对共享内存进行操作的代码编写就需要额外小心谨慎，稍有不慎就会造成严重的后果。线程间的通信很简单的，但是线程同步过程中所发生的错误是很难检测，重现以及修复的。

过度的上下文切换

当CPU从执行一个线程切换到执行另外一个线程时，CPU需要保存当前线程的本地数据，程序计数器等等，然后将即将执行的线程的本地数据和程序计数器等数据载入，这个切换称为“上下文切换”。CPU从一个线程的执行上下文切换到另一个线程的上下文。

上下文切换代价很大，你肯定不希望在非必要情况下进行线程上下文的切换。

你可以在维基百科上获得更多关于上下文切换的知识：

(http://en.wikipedia.org/wiki/Context_switch)

资源消耗的增加

一个线程运行时需要很多计算机资源，除了CPU运行时间外还需要内存来存放本地栈。除此之外，操作系统也需要消耗一些计算资源来管理这个线程，试试写一个程序在运行时创建1000条线程，这1000条线程什么都不做，但又不结束，然后去查看当程序运行时内存的占用情况。

[译]java多线程与同步机制系列(2)－－java多线程的优势

By 朱家骅

发表于 2015-04-23

原文链接(http://tutorials.jenkov.com/java-concurrency/benefits.html)


尽管多线程会带来各种各样的困扰和挑战，但是java多线程开发依然流行，其原因是因为它具有以下优点：

  1. 更好的资源利用
  2. 某些场景下简化程序设计
  3. 使程序更具响应性

更好的资源利用

想象一个从本地文件系统中读取和处理文件的程序，我们假设读取一个文件需要5秒钟，处理一个文件需要2秒钟，那么处理2个文件就需要花费14秒钟：

5 seconds reading file A
2 seconds processing file A
5 seconds reading file B
2 seconds processing file B
-----------------------
14 seconds total

当从磁盘上读取文件的时候，CPU此时便在等待读取文件完成然后继续相关操作，这也就是说这一段时间CPU其实是浪费掉了，它本应该可以执行其他程序。当然，通过调整执行顺序，我们可以优化CPU的执行效率，接下来我们来看看如何优化：

5 seconds reading file A
5 seconds reading file B + 2 seconds processing file A
2 seconds processing file B
-----------------------
12 seconds total

当CPU在等待文件B读取完成的时间内，可以处理文件A，这样，CPU就不会被浪费。记住一点：当CPU在等待文件读取的时候它是出于空闲状态的！！！


总之，CPU在等待IO的时候是可以执行其他任务来避免浪费的，不只是磁盘IO，在等待网络IO以及用户输入的时候也是如此。另外记住一点，磁盘IO和网络IO是比CPU与内存之间的IO慢得多的。

更简单的程序设计

如果你是在一个单线程程序中按照以上的读取顺序运行的话，你可以跟踪和控制每个文件的读取和处理状态。但是如果创建两个线程，每个线程负责其中一个文件的读取和处理的话，问题就来了，两个线程中总有一个会在发起文件读取操作时被阻塞，因为此时另外一个线程正在进行IO操作，那么当这个线程在等待的时候，另一个线程就可以占用CPU来处理已经读取到的文件，那么结果就是磁盘就一直处于忙碌状态，不断的将文件从磁盘中传输到内存中。多线程的使用使得CPU和磁盘的效率都有所提升，结果就是整个程序效率的提升。程序设计的简单体现在每个线程只需要处理一个文件。

响应性

多线程程序相比较单线程程序更具响应性，想象一个从多个端口接受请求的服务器程序，当一个请求到来的时候，它处理完请求后返回继续监听端口，整个服务器的运行框架如下：

一种可行的改进方案时监听线程将接受到的请求传递给一个工作线程处理然后继续监听，工作线程处理请求后将结果返回给客户端，这种设计的框架如下：

这种模式对桌面程序同样有效。当点击一个按钮开始执行一个长时间任务时，如果是采用单线程，那么执行这个长时间任务的线程就是UI线程，那么此时这个程序给用户造成假死的表象，而实际上，此时程序正在执行一个用户主动请求的任务，这就是不具备响应性的表现。换一种方式，像上面服务器程序一样，把这个长时间任务交给工作线程来执行，UI线程依然能够处理用户的其他请求，工作线程执行完后，通知UI线程更新UI给予用户结果提示，这种依靠工作线程执行复杂耗时任务，UI线程处理用户UI操作的模式更具有我们所说的响应性。

Java多线程
 

[译]java多线程与同步机制系列(1)－－综述

By 朱家骅

发表于 2015-04-23

原文链接 [http://tutorials.jenkov.com/java-concurrency/index.html]


早在单核时代，计算机只能同时执行一个进程。随后，多任务还是兴起，这意味着计算机可以“同时”执行多个进程，当然，这里的同时不是真正的同时，只是单核CPU被进程轮流使用，操作系统控制进程轮流获取CPU控制权，每个进程执行一段时间片后让出CPU给另一进程使用。


多任务的兴起也给开发者带来了挑战，程序将不再被认为可以一直拥有包括CPU，内存以及其他计算资源了，一个行为良好的进程应该在不需要某些资源时及时释放，以便让其他进程使用。


多线程技术的出现意味着在同一个进程中可以有多个线程并发执行任务，线程成为了计算机运行的最小调度单位，一个线程的执行对应一个CPU的执行，当你在程序中使用了多线程时，就好像在同一个程序中有多个CPU在执行任务。


多线程对某些程序的性能及效率提升有很大的帮助，但是，多线程是一门比多任务更具有挑战性的技术。同一个进程中的多个线程共享同一块内存，由此会出现在单线程程序中不会出现的同时读写内存问题。在单线程程序中，这些错误永远都不会出现，因为两个线程永远不会同时执行。由于现代计算机一般都配备了双核甚至多核CPU，这就意味着多个线程可以同时在多个CPU中执行，这从一定程度上缓解了多线程所带来的困扰。





当一个线程在读取某个内存单元的值时，另外一个线程同时在向该内存单元写入新值，那么第一个线程将读取的是原值还是另外一个线程写入的新值，或者两者都不是呢？又或者两个线程同时对一个内存单元写入新值，那么最后该内存单元存储的到底是哪个线程写入的值抑或是一个奇怪的值呢？


如果没有采取一定的措施的话，以上的情况都有可能发生，更要命的是这些情况是不可预测的，每次运行结果都会不一样。因此对于开发人员来说，采取合理的措施来控制多线程对其共享的内存资源，文件，数据库的操作是必不可少的，而这也将是本次java同步及多线程系列教程的重点之一。

java多线程及同步

java是率先具备多线程能力的少数几门编程语言之一，开发者可以利用java快速开发多线程程序，因此，java开发者也会遇到上面提到的一系列问题。这也就是我为什么写这篇java教程的原因，不仅仅是自我的总结，同时也将使得像我一样的java开发者从中受益。


这篇教程原本是准备只讲java多线程的，但是多线程中经常出现的问题与分布式系统中的多任务出现的问题非常相似，因此，我干脆把分布式系统中的多任务也加进了教程讲解，所以主旨也从多线程变成了同步。

java同步的发展趋势

自从第一本讲解java同步机制的书面世以来，世界上出现了许多同步架构和设计，甚至java5也发布了同步组件。


现如今，像Vert.x,Play/AKKA,Qbit这样的异步独享的平台和API开始兴起，这些平台使用了与Java／JEE不同的线程同步模型，共享内存模型以及锁模型。同时，非阻塞的同步算法以及非阻塞的工具，比如LMax Disruptor等也可以被开发者所用。java7中Fork and Join框架也新增了一种函数式串行编程模式，java8中也出现了stream API。


正是因为在java同步技术有如此多的发展促使我开始不得不更新我之前的java同步系列教程。因此，这个教程的编写又被提上了日程，我将利用空闲时间尽快更新并发布，敬请期待！！！
