wait(),notify(),notifyAll()

忙等待对于运行该等待线程的CPU来说不是一个对CPU有效利用的方式，除非线程平均等待时间非常的短。另外，如果等待线程在信号到来之前一直处于休眠或者非活跃状态的话，那就使得线程间通信变得更加智能。

java内置的线程等待机制可以使线程在等待时进入非活跃状态，java.lang.Object类定义了三个方法wait(),notify(),notifyAll()实现了该机制。

线程调用对象的wait()方法的话会进入非活跃状态直到其他线程调用对象的notify()等待线程重新变成活跃状态。为了能够调用对象的wait()方法或者notify()方法，执行线程必须获得该对象的锁。换句话说，调用线程必须在一个同步块里面调用wait或者notify方法才能有效。下面就是MyWaitNotify类，这是一个MySignal的变体，它使用了wait()方法和notify()方法。

等待线程调用doWait方法进入等待状态，通知线程调用doNotify方法激活等待线程。另外还有一个notifyAll方法将激活所有该对象上的等待线程。

正如你所看见的，线程都是在同步块中调用wait方法和notify方法，这是必须的！！！线程不可以在没有获得该对象的锁的情况下调用wait，notify，notifyAll方法。如果没有在同步块中调用这些方法，则会抛出一个IllegalMonitorStateException异常。

但是，这是怎么做到的呢？等待线程在执行同步块的时候难道没有获得对该对象的锁？如果获得了该对象的锁的话，通知线程怎么可以进入到同步块中去调用notify方法呢？
答案是当一个线程调用wait方法后他便释放了对该对象的锁，这样就允许其他线程调用wait或者notify方法了，因为这些方法必须在同步块中调用。

一旦线程被唤醒了它也无法立即退出wait方法调用直到调用notify的线程退出同步块。换句话说，被唤醒的线程必须重新获得对该对象的锁才能退出wait调用，因为wait调用是在同步块中进行的。如果多个线程通过nptifyAll方法被唤醒的话，那么一次只有一个线程可以退出wait方法调用，因为每个线程都必须获得该对象的锁才能退出wait调用。

信号丢失

notify和notifyAll不会保存调用它们的方法，有可能此时根本没有等待线程，那么通知信号就被白白丢弃了。因此，如果线程在其他线程调用wait方法前调用notify方法的话，那么这个信号就会被等待线程错过。而这将有可能会是等待线程永远都处于等待状态，无法被激活，因为通知信号已经丢失。

多个线程等待同一个信号

如果有多个线程在等待信号，并且被notifyAll方法一次性激活的话，使用while循环也是一种不错的解决方案。由于只允许一个等待线程继续执行，意味着只有一个线程重新获得该对象的锁，退出wait调用并将wasSignalled标志清空，一旦该线程退出了doWait方法中的同步块，其他线程就可以获得锁并且退出wait调用，然后检查wasSignalled变量的值，由于前一个线程已经把wasSignalled对象值清空了，所以剩下的线程又会调用wait方法进入等待状态，等待下一个信号到来。

这看上去不会出现大问题，毕竟即使线程A或线程B被MyWaitNotify2唤醒的话，它们都会通过while循环查看信号状态，由于MyWaitNotify1对象的doNotify方法没有被调用，所以它们会执行wait方法继续等待。这种情况就是上面提到的虚假唤醒，线程A和线程B都没有被信号唤醒，但是while循环可以解决这个问题，两个线程又重新回到等待状态了。

但是问题是，doNotify方法调用的时notify方法而不是notifyAll方法，那么线程A，B，C，D中只有一个可以被唤醒。所以，如果线程A或B被本来应该唤醒线程C或D的信号所错误唤醒的话，那么线程A或B将通过while循环查看信号状态，由于myWaitNotify1的doNotify方法没有被调用，所以wasSignalled为false，那么线程A或B将再次执行wait方法进入等待状态，而线程C或D没有机会去查看本来就是用来激活它们的信号，也就是说，信号丢失了。这种情况就是上面提到的信号丢失。线程C或D错过了发给它们的信号。

如果doNotify方法中使用的是notifyAll方法而不是notify方法的话，所有的线程都将会被唤醒并检查信号，那么即使线程A或线程B被错误唤醒，但是它们检查信号时发现没有信号，那么它们都会重新回到等待状态，而线程C或D激活后查看信号，发现有信号则将继续执行，另外一个线程则重新回到等待状态，因为前一个线程已经将信号清空了。

你可能会想着以后直接用notifyAll来激活线程而不是notify方法，但是这是不明智的，因为如果只有一个线程可以对信号产生响应就没有任何理由来唤醒所有的线程。

所以一定要记住，不要在全局对象，String常量上使用wait/notify/notifyAll方法。监视器(即被加锁的对象)必须对于它所属的对象是独立的。就像上面的MyWaitNotify3类一样，每个该类的对象都持有一个自己的MonitorObject对象，而不会在空的String常量上调用wait或notify方法。

java volatile关键字用来修饰一个变量使其最新值能一直保存在内存中，更精确的讲是线程每次都直接从内存中读取该变量的值而不是CPU缓存，线程每次都将该变量的新值直接写到内存而不仅仅是CPU缓存。

事实上，java 5新增的volatile关键字不仅仅只保证线程从内存直接读取或写入数据，我在下面会细说。

java volatile保证变量的可见性

volatile关键字可以保证跨线程间的变量可见性，这听起来可能有点抽象，下面让我来举个例子。

在一个多线程的应用程序中，当多个线程操作一个非volatile的变量时，每个线程都会将该变量读取到CPU缓存中，基于提升效率的要求，现代计算机都是多CPU的架构，这就意味着每个线程可能在不同的CPU上执行，那么每个线程就将该非volatile变量的值读取到不同的CPU缓存中，如下图所示：

想象一下两个或多个线程对包含一个counter变量的对象进行访问：

下面这种情况下，当多个线程读写一个变量时，仅仅声明volatile变量是不够的。线程1将counter从内存中读取到CPU缓存中，此时counter值为1，就在这个操作后，线程2立即也将counter读取到它所属的CPU缓存中，注意这个时候counter值仍然为0，现在线程1和线程2都将counter加1并且立即写回到内存中，那么内存中counter的值将是1，而期望的值应该是2.

多线程中这个问题叫做可见性问题，线程读取了一个被其他线程修改但是还没有及时写回到内存中的变量，也就是说线程对变量的修改对其他线程不可见。

java volatile的其他作用

java 5新增的volatile关键字不仅仅只是保证线程直接从内存读取变量或者直接将变量新值回写到内存中，它还有以下作用：

  1. 如果线程A将新值写入volatile的变量，然后线程B读取该变量，那么在线程A写入变量之前的所有变量对线程B是可见的。
  2. volatile变量的读写字节码指令不会被java虚拟机重排序(java虚拟机为了提升程序运行效率，会在不会对程序结果造成影响的情况下对java字节码指令进行重排序。译者注：这涉及到java字节码的相关知识，博主以后可能会发布关于java字节码的相关内容)。在操作volatile变量之前和之后的字节码指令可以被重排序，但是volatile变量的读写指令不会和其他指令混合，意思是读写volatile变量之后的字节码指令即使被重排序它们仍然还是在读写volatile变量之后执行。

何时使用volatile就足够了

根据我上面提及的那样，如果多个线程访问共享变量的话仅仅使用volatile是不够的，你需要使用synchronized机制来保证对变量的读写是原子性的。

但是如果是一个线程对变量进行读取和修改，其他线程只是读取这个变量，那么读取线程就可以保证获取的是变量的最新值，这就需要volatile关键字的保证。(译者注：这个地方我有一个疑问，想象一下这种情况，线程A读取变量，在修改该变量之前，其他线程也从内存中读取该变量，这个变量的值仍然为原值，而不是被线程A修改后的值。这个地方可能就是看具体的业务需求了，如果要求是线程获取的必须是最新值的话，那么线程A修改了变量值后，之前读取线程读取的值就是旧值了，那么这种情况下仅仅靠volatile也是不够的)

volatile的性能问题

对volatile变量的读取和修改都直接与内存交互，同时我们也知道，CPU对内存的访问效率比直接访问CPU缓存慢太多，这就带来了效率问题。同时上面也提到对volatile变量的访问的字节码指令禁止重排序，而重排序是提升程序运行效率的重要手段。因此综合来看，最好是在确实需要保证变量可见性的前提下使用volatile关键字。

java内存模型详细阐述了java虚拟机在计算机内存(RAM)中的工作原理。java虚拟机可以算作是一个完整的计算机模型，所以它也具有自己的内存模型，也就是我们所说的java内存模型。

如果你想设计一个行为良好的同步程序的话，java内存模型是必须掌握的知识。java内存模型阐述了不同线程如何获得其他线程对共享数据的修改，以及在必要时如何同步各个线程对共享数据的访问。

java最初的内存模型是不完善了，所以java在1.5版本的时候对java内存模型做了大幅度的修改更新，这个新版本直到现在的java8都在使用。

java内存模型内部结构

java虚拟机中的java内存模型将内存分为了线程栈和堆。下图展示了这种结构：

在java虚拟机中每个线程都有自己的线程栈，线程栈包含了该线程当前正在执行的方法的信息，我更愿意把它叫做方法调用栈，随着线程的执行，线程栈的内容也在不断变化。

线程栈中也包含了当前执行方法的本地变量，线程只能访问属于它自己的线程栈，所以由该线程创建的本地变量对其他线程是不可见的。即使两个线程在执行同一段代码，它们创建了相同名字的本地变量，但这个变量都保存在它们各自的线程中。

所有基本类型的本地变量都是直接存储在线程栈中，因此这些本地变量在线程之间是不可见的。一个线程可以把一个基本类型本地变量的拷贝传给另一个线程使用，但是无法将其自身传给其他线程使用。(译者注：java中，基本数据类型是值传递，非基本类型是引用传递)

堆中存放了java程序中创建的所有非基本类型对象(译者注：基本类型可以被封装成类对象，这个时候它就存放在堆中。比如int - 》 Integer)，不管对象是被创建然后将对象引用传给方法的本地变量，还是作为其他对象的成员变量，它都将存放在堆中。

下图就是调用栈，线程栈中的本地变量以及堆中存放的对象示意图：

本地变量如果是一个基本数据类型的话，那么它就被存放在线程栈中。

如果本地变量是一个对象引用。那么这个引用是存放在线程栈中，但是它所引用的对象是存放在堆中。

一个对象可能会包含很多对象方法，这些方法中的本地变量都将被存放在线程栈中，即使这个方法所属的对象被存放在堆中。

对象的成员变量都将随对象一起存放在堆中，不管这个成员变量是基本数据类型还是一个对其他对象的引用。

静态类变量和类的其他信息都存放在堆中。

硬件存储架构

现代硬件存储结构和java内存模型有很大的区别，了解硬件存储架构对于理解java内存模型如何在其基础上工作是非常重要的。这一节先讲通用的硬件存储架构，然后再将java内存模型在这个架构基础上是如何工作的。

下图是现代计算机的硬件架构：

一台现代计算机一般有两个或多个CPU，每个CPU上可能有多个核，这就意味着可以有超过一个线程同时运行，每个CPU都可以运行一个线程。如果你的java程序是多线程的，那么在程序内部可以有多个线程同时运行。

每个CPU都有许多寄存器，CPU操作寄存器的数据的速度比操作内存数据的速度快得多，因为CPU访问寄存器的速度比访问内存的速度快得多。

每个CPU也可能拥有一个CPU缓存层。事实上，现代CPU都会有一个一定大小的缓存层。CPU访问缓存层的速度比访问内存的速度快，但是比访问寄存器的速度慢。有些CPU甚至有多个缓冲层(Lay1和Lay2)，当然，在这里我们不必了解java内存模型与这些缓冲层是怎么打交道的，我们只需要知道CPU也有自己的缓冲层存储数据。

计算机也有主存(RAM)，这是我们最熟悉的。CPU可以访问主存，主存的的容量比CPU的缓存大多了。

当CPU需要访问主存时，它一般将一部分主存的数据直接保存到它的缓存中，甚至会将一部分数据直接存到CPU的寄存器中，然后对这些数据执行操作。当CPU需要将结果回写到主存时，它首先把寄存器中对应的值回写到CPU缓存中，然后从缓存中将值回写到主存中。

CPU一般会在需要缓存其他数据时从缓存中清除一部分数据并刷新内存中对应的数据。CPU缓存可以一次对部分缓存区域进行操作而没必要每次对整个缓存进行操作。一般来说，CPU从内存读取数据或从缓存中清除数据的最小单位叫做“缓存条“(cache lines)，CPU可以一次将一个或多个缓存条的数据读进缓存，也可以一次将一个或多个缓存条清除出缓存。

java内存模型与硬件存储模型结合

前面我们已经提到，java内存模型与硬件存储模型有很大的区别。硬件存储模型不区分线程栈和堆。从硬件层面上看，线程栈和堆都存放在主存中。部分线程栈和堆有可能会被CPU加入到缓存中甚至是寄存器中。下图显示了java内存模型与硬件存储模型的联系：

由于对象和变量可以存放在不同的存储区域，这样会带来很多问题，其中最主要的两个问题是：

  1. 线程对共享数据的修改的对其他线程的可见性。
  2. 当访问共享数据时的竞争问题。


共享对象的可见性

比如说一个共享对象最开始存储在主存中，正在一个CPU上执行的线程将该对象读取到CPU缓存中然后修改了这个对象的值，那么只要CPU没有将缓存中被修改的对象刷新到主存中的话，那么其他访问该对象的线程访问的就是过时的数据，这就造成了可见性问题。最坏的情况是每个线程都持有一个该对象的拷贝，每个对象拷贝都存放在不同的CPU缓存中。

下图的例子就是这种情况。左边的CPU上的线程将一个共享对象的拷贝存放在CPU缓存中，然后修改了该对象的count属性值为2，这个修改对右边CPU上的线程是不可见的，因为这个新修改的对象没有被刷新到主存中对应的对象。

为了解决这个问题，可以给count变量加上volatile关键字，这个关键字可以保证count变量每次都从主存读取，然后修改之后立即写回到主存中。

竞争

如果多个线程共享同一个对象，并且同时对该对象执行更新操作，这个时候就会发生竞争问题。

比如线程A将主存中的一个对象读取到CPU缓存中，然后线程B也将该对象读取到另一个CPU缓存中，现在线程A对count变量加一，线程B同样也对count变量加一。这样count变量的值在两个CPU缓存中分别加了一。如果这些线程是顺序执行的话，那么两个线程执行完后主存中对象的count变量值将加二。但是如果两个线程是同时执行的话，那么不管是线程A还是线程B将count新值更新到主存中，最后主存中count的值都只加一，虽然总共被加了两次。

判断你的代码是否是线程安全的方法是判断它是否遵守线程控制逃逸原则，原则如下：

If a resource is created, used and disposed within
the control of the same thread,
and never escapes the control of this thread,
the use of that resource is thread safe.

这里的资源包括对象，数据，数据库连接等等，在java中你可能永远不会显式的“抛弃“资源，这里所说的“抛弃”是指将对象的引用置空(null)；

当然，即使这个对象引用的使用是线程安全的，但是这个引用所指向的资源比如说文件，数据库连接等等确有可能不是线程安全的。举个例子，如果线程一和线程二分别创建了一个数据库连接的话，那么每个线程对该连接的使用是线程安全的，但是连接所指向的数据的使用却不是线程安全的。比如两个线程执行以下操作：

check if record X exists
if not, insert record X

如果两个线程同时执行以上操作的话，那么X的状态对于两个线程是相同的，那么就有可能两个线程都会插入一个X，总共就是两个X记录了：

Thread 1 checks if record X exists. Result = no
Thread 2 checks if record X exists. Result = no
Thread 1 inserts record X
Thread 2 inserts record X

这种情况也有可能发生在操作文件或者其他共享资源的时候。所以区分线程中的对象是一个资源还是仅仅是对资源的引用非常重要。

Spring Boot

最后还想说说Spring Boot，这是我最近接触到的一个新框架，感觉这个框架用处很大，所以在这里说说。Spring boot是Spring官方的一个子项目，按照官方的说法，Spring Boot能够让你快速开发部署用于生产环境下的Spring项目，它遵循约定大于配置的原则。关于约定大于配置的原则，我在这里多说几句，到底什么是约定大于配置，简单的来说就是这个框架有一个规范，比如说配置文件放在哪个文件夹下，取什么名字，配置文件的格式规范是什么样子，这叫做约定。当然这个约定是可以不遵守的，你完全可以自定义，这叫做配置。约定大于配置的意思就是希望开发者按照它的规范来使用这个框架，而不是自定义配置，这个原则在很多框架上都是被强调的。这么做有两点好处：

  1. 加快框架初始化和执行速度；框架在初始化时需要根据配置文件的内容对需要用到的类进行初始化，那么它将首先在规范中规定的配置文件位置去寻找配置文件，如果按照规范来，那么框架就一下子找到了配置文件，如果按照自定义配置来的话，框架先得遍历规范中的位置然后再到配置的文件路径去寻找，这无疑拖慢了速度。
  2. 减少开发者的错误几率；一个框架往往有很多东西需要配置，如果开发者自己定义这些配置方式难免会犯一些错误，有些错误可能会影响整个框架的运行。如果根据规范来做的话，犯错可能性会降低许多。

那么Spring Boot跟这篇文章有什么关系呢？答案是Spring Boot提供了内置的tomcat服务器程序，不仅如此，还包括了像Jetty，Undertow这些很流行的服务器，这使得我们可以无需手动部署war到服务器上。还有一点也是Spring Boot带给我们的便利，那就是我们可以不用XML配置，直接使用java代码或注解写出一个纯java的应用。当然，要想使用Spring Boot的话需要一些配置，这里以maven为例：

继承还是实现？

我们没有办法判断这两种方式孰优孰劣，因为它们都能很好的实现我们的需求。但是从我个人看来，我更倾向于使用第二种方法，也就是通过实现Runnable接口来写入业务代码。如果我们使用到了线程池的话，那么可以将所有需要执行的Runnable对象组成一个队列，一旦线程池中出现了空闲线程的话，就可以从队列中拿出一个Runnable对象给该空闲线程去执行，而如果采取第一种方式即子类化Thread，就没法像第二种方式那么灵活。


在某些情况下，你有可能需要同时使用这两种方式。比如说创建一个能执行多个Runnable的线程子类对象，而这就是线程池的设计原理。
