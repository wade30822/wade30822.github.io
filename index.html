原文链接(http://tutorials.jenkov.com/java-concurrency/read-write-locks.html)

读写锁是一种比上篇博客中提到的锁更加复杂的锁实现。想象一下你的应用需要对某些共享资源进行读写，并且读操作比写操作频繁的情况，两个线程读取同一个资源对双方都不会造成影响，所以多个线程可以同时对同一个资源进行读操作，但是如果一个线程需要对资源进行写操作，那么其他线程就无法对其进行读或者写操作。为了使得资源可以被多个资源同时读但是只允许一个线程写，就需要一个读写锁。

java5中java.util.concurrent包里为我们提供了读写锁的实现，我们可以在需要它的时候直接使用，但是了解其背后的实现原理对我们来说也是很有必要的。

java中读写锁的实现

首先我们总结一下获取资源读和写权限的情形：

  1. 读权限： 没有线程获得该资源的写权限或者没有线程请求获得该资源的写权限
  2. 写权限： 没有线程获得该资源的读权限或者写权限

如果一个线程想获得资源读权限，只要当前没有线程正在对资源进行写操作或者没有线程请求获得该资源的写权限的话，该线程可以成功获得读权限，在这里之所以优先考虑线程对资源的写权限请求是因为我们假设写操作比读操作更加重要。另外，如果读操作经常发生并且我们没有优先考虑写操作的话，就有可能发生饥饿。请求写操作权限的线程就一直阻塞直到所有读线程解锁该Lock对象。如果新的线程一直获取读权限的话，那么申请写权限的线程将一直阻塞导致饥饿。为了解决这个问题，线程只有在没有其他线程在执行写操作或者请求写操作权限的情况下才能获取该资源的读操作权限。

线程想要获取写权限的话必须保证没有其他线程正在对该资源进行读操作或者写操作。在这里，除非你希望在请求写权限的线程间保持公平性，否则我们不会理会到底有多少线程在请求写权限或者以什么顺序在请求获取写权限。

在ReadWriteLock对象上有多个线程在等待获取读权限，多个线程在等待获取写权限。如果一个等待获取读权限的线程被notify唤醒了，那么该线程又将进入等待状态因为此时有多个线程在等待写权限，那么这次notify等于没有任何效果，没有任何线程可以获得读权限或者写权限。所以这里我们使用notifyAll方法唤醒所有等待线程，这样总会有线程可以获取到它想要的权限。

使用notifyAll还有一个好处，当多个线程在等待读权限并且没有线程在等待写权限时，unlockWrite方法的调用会唤醒所有线程，那么等待读权限的线程可以同时获取读权限而不必一个接一个的被唤醒获取读权限。

可重入的读写锁

上面这个ReadWriteLock锁的实现不是可重入的，如果一个已经持有写权限的线程再次请求写权限的话将陷入阻塞因为此时已经有一个线程获取了写权限，而这个线程就是它自己。另外我们再考虑以下这种情况：

  1. 线程1持有读权限
  2. 线程2请求写权限但是被阻塞，因为此时线程1持有读权限
  3. 线程1再次请求读权限但是被阻塞，因为现在有一个线程2的写权限的请求。

上面这种情况有点类似于死锁，此时没有任何线程可以获取读权限或者写权限。

为了让ReadWriteLock成为可重入的，我们需要对代码进行一些修改。我们将分开讨论针对读操作和写操作的可重入问题。

Slipped Conditions就是线程在执行操作前需要检查一个状态值，但是这个状态值却被其他线程修改而导致该线程执行出错。下面是一个简单的例子：

lock方法有两个同步块，第一个同步块的退出条件为isLocked为false，第二个同步块是通过将isLocked设置为true来防止其他线程获取当前Lock对象的锁。

想象一下如果isLocked为false，两个线程同时调用lock方法，当线程1进入同步块中，它检查isLocked的值为false，这样它便退出同步块，这样线程2开始进入同步块，此时isLocked的值同样为false，线程2退出同步块。这样两个线程都将都将进入第二个同步块，将isLocked设置为true。

这种情况就叫做slipped conditions。所有的线程都检查一个状态值，然后退出同步块在当前线程没有修改状态值之前让其他线程检查该状态值。换句话说就是状态从被线程检查到被线程修改期间已经被其他线程修改过了。

为了避免slipped conditions，对状态的检查和设置必须对于执行该操作的线程来说是原子性的，这也就意味着其他线程无法在当前线程检查和设置状态期间访问该状态。

解决方法很简单，就是把isLocked = true；这条语句放到第一个同步块中while循环之后即可。就像下面这样：

//Fair Lock implementation with nested monitor lockout problem

public class FairLock {
private boolean isLocked = false;
private Thread lockingThread = null;
private List waitingThreads =
new ArrayList();

public void lock() throws InterruptedException{
QueueObject queueObject = new QueueObject();

synchronized(this){
  waitingThreads.add(queueObject);

  while(isLocked || waitingThreads.get(0) != queueObject){

    synchronized(queueObject){
      try{
        queueObject.wait();
      }catch(InterruptedException e){
        waitingThreads.remove(queueObject);
        throw e;
      }
    }
  }
  waitingThreads.remove(queueObject);
  isLocked = true;
  lockingThread = Thread.currentThread();
}

当一个已经对Lock上锁的线程尝试unlock时，它将在进入unlock方法的synchronized(this)代码块时被阻塞并将一直等待直到lock方法中的等待线程离开synchronized(this)代码块。但是lock方法中的等待线程只有当isLocked为false时才会离开synchronized代码块，而这个操作需要在unlock方法中完成。

简而言之就是说，lock方法中的等待线程需要某个线程执行unlock方法才能使等待线程退出lock方法，但是没有任何线程可以在等待线程退出lock方法前执行unlock方法。

如果CPU被其他线程占用而导致某个线程一直无法获得CPU执行时间，这就叫做饥饿。解决饥饿的办法叫做公平锁，意思是所有的线程都会获得执行的机会。

造成饥饿的原因

在java中以下三点会导致线程饥饿：

  1. 高优先级的线程占用了所有的CPU执行时间。
  2. 线程在等待进入同步块时被无限期的阻塞，因为其他线程总是先比它进入同步块。
  3. 线程无限等待一个对象锁，因为其他线程一直被唤醒获取该对象锁。

高优先级的线程占用了所有的CPU执行时间

你可以单独为某个线程设置一个优先级，高优先级的线程将获得更多的CPU执行时间。线程优先级的范围是1到10，具体的调度执行取决于当前操作系统。对于大多数程序而言最好不要修改线程的优先级。

线程无限期的等待进入同步块

java同步块是造成线程饥饿的另一个原因。java同步块无法保证等待线程进入同步块的顺序。这意味着存在理论上的风险即某个线程一直无法进入同步块因为其他线程一直获得权限进入。这就造成了线程饥饿，线程将由于其他线程一直获得CPU执行时间而”饥饿至死”。

线程无限期等待一个对象锁

在多个线程等待同一对象锁的情况下，notify方法没有规定哪个线程先被唤醒获取锁，也就是说任何一个等待线程都有可能被唤醒。因此这就存在其中一个线程一直无法被唤醒的风险，因为其他线程一直被唤醒而不是它。

锁排序是一个简单有效的死锁预防机制。但是它的前提是你需要知道在获取一个锁前你需要先获取哪些锁，而这不是一件简单的事。

另外一个死锁预防机制是对线程获取锁的过程设置超时时间，如果在这段时间之内没有获取到锁的话就将放弃。如果一个线程在规定时间内没有获取锁的话，该线程将做好备份并且释放所有已经获取的锁，然后等待一段随机时间后重新尝试获取。这个随机等待时间也给了其他线程一个机会去获取由该等待线程所释放的锁，这可以确保程序能够继续执行下去。

死锁检测

死锁预防是在锁排序和锁超时很难实现或效果不好的情况下采用的复杂死锁预防机制。

当线程获取到某个锁后，一般会记录在线程和锁对象的数据结构中(map,graph等等)。除此之外，当线程请求锁时，这个请求也会被记录在对应线程和锁对象的数据结构中。

当线程请求一个锁时被拒绝，那么该线程将遍历锁的记录来检查死锁。比如说，线程A请求锁7，但锁7被线程B持有，然后线程A将检查线程B是否需要线程A已经持有的锁，如果是，那么发生死锁。

当然死锁发生的情况会比两个线程各自持有对方需要的锁更加复杂。线程A可能等待线程B释放锁，线程B等待线程C释放锁，线程C等待线程D释放锁，线程D等待线程A释放锁。如果线程A需要检测是不是发生了死锁，那么它将遍历检查线程B请求的所有锁，然后根据这些锁检查线程C,然后线程D，随后它从线程D的请求锁中找到了自己持有的锁，那么它就知道发生死锁了。

线程信号的的作用是线程与其他线程之间发送信号来通信。除此之外，线程信号可以让一个线程等待另一个线程给它发送信号。比如线程B可能需要等待线程A给它发送一个信号表明线程B需要的数据已经准备好了。

一种线程间最简单的发送信号的方法就是通过给共享对象的某些成员变量设置信号值。线程A可能会在一个同步块中将hasDataToProcess的值设为true，随后线程B可能会在一个同步块中读取hasDataToProcess的值来判断是否数据准备好了：

wait(),notify(),notifyAll()

忙等待对于运行该等待线程的CPU来说不是一个对CPU有效利用的方式，除非线程平均等待时间非常的短。另外，如果等待线程在信号到来之前一直处于休眠或者非活跃状态的话，那就使得线程间通信变得更加智能。

java内置的线程等待机制可以使线程在等待时进入非活跃状态，java.lang.Object类定义了三个方法wait(),notify(),notifyAll()实现了该机制。

线程调用对象的wait()方法的话会进入非活跃状态直到其他线程调用对象的notify()等待线程重新变成活跃状态。为了能够调用对象的wait()方法或者notify()方法，执行线程必须获得该对象的锁。换句话说，调用线程必须在一个同步块里面调用wait或者notify方法才能有效。下面就是MyWaitNotify类，这是一个MySignal的变体，它使用了wait()方法和notify()方法。

等待线程调用doWait方法进入等待状态，通知线程调用doNotify方法激活等待线程。另外还有一个notifyAll方法将激活所有该对象上的等待线程。

正如你所看见的，线程都是在同步块中调用wait方法和notify方法，这是必须的！！！线程不可以在没有获得该对象的锁的情况下调用wait，notify，notifyAll方法。如果没有在同步块中调用这些方法，则会抛出一个IllegalMonitorStateException异常。

但是，这是怎么做到的呢？等待线程在执行同步块的时候难道没有获得对该对象的锁？如果获得了该对象的锁的话，通知线程怎么可以进入到同步块中去调用notify方法呢？
答案是当一个线程调用wait方法后他便释放了对该对象的锁，这样就允许其他线程调用wait或者notify方法了，因为这些方法必须在同步块中调用。

一旦线程被唤醒了它也无法立即退出wait方法调用直到调用notify的线程退出同步块。换句话说，被唤醒的线程必须重新获得对该对象的锁才能退出wait调用，因为wait调用是在同步块中进行的。如果多个线程通过nptifyAll方法被唤醒的话，那么一次只有一个线程可以退出wait方法调用，因为每个线程都必须获得该对象的锁才能退出wait调用。

信号丢失

notify和notifyAll不会保存调用它们的方法，有可能此时根本没有等待线程，那么通知信号就被白白丢弃了。因此，如果线程在其他线程调用wait方法前调用notify方法的话，那么这个信号就会被等待线程错过。而这将有可能会是等待线程永远都处于等待状态，无法被激活，因为通知信号已经丢失。

多个线程等待同一个信号

如果有多个线程在等待信号，并且被notifyAll方法一次性激活的话，使用while循环也是一种不错的解决方案。由于只允许一个等待线程继续执行，意味着只有一个线程重新获得该对象的锁，退出wait调用并将wasSignalled标志清空，一旦该线程退出了doWait方法中的同步块，其他线程就可以获得锁并且退出wait调用，然后检查wasSignalled变量的值，由于前一个线程已经把wasSignalled对象值清空了，所以剩下的线程又会调用wait方法进入等待状态，等待下一个信号到来。

记住，即使四个线程对同一个对象调用wait或者notify方法，但是信号都只是被保存到单独的MyWaitNotify对象中，对MyWaitNotify1对象调用doNotify方法可能会激活MyWaitNotify2对象上的等待线程，但是信号只是被保存在MyWaitNotify1对象中。

这看上去不会出现大问题，毕竟即使线程A或线程B被MyWaitNotify2唤醒的话，它们都会通过while循环查看信号状态，由于MyWaitNotify1对象的doNotify方法没有被调用，所以它们会执行wait方法继续等待。这种情况就是上面提到的虚假唤醒，线程A和线程B都没有被信号唤醒，但是while循环可以解决这个问题，两个线程又重新回到等待状态了。

但是问题是，doNotify方法调用的时notify方法而不是notifyAll方法，那么线程A，B，C，D中只有一个可以被唤醒。所以，如果线程A或B被本来应该唤醒线程C或D的信号所错误唤醒的话，那么线程A或B将通过while循环查看信号状态，由于myWaitNotify1的doNotify方法没有被调用，所以wasSignalled为false，那么线程A或B将再次执行wait方法进入等待状态，而线程C或D没有机会去查看本来就是用来激活它们的信号，也就是说，信号丢失了。这种情况就是上面提到的信号丢失。线程C或D错过了发给它们的信号。

如果doNotify方法中使用的是notifyAll方法而不是notify方法的话，所有的线程都将会被唤醒并检查信号，那么即使线程A或线程B被错误唤醒，但是它们检查信号时发现没有信号，那么它们都会重新回到等待状态，而线程C或D激活后查看信号，发现有信号则将继续执行，另外一个线程则重新回到等待状态，因为前一个线程已经将信号清空了。

你可能会想着以后直接用notifyAll来激活线程而不是notify方法，但是这是不明智的，因为如果只有一个线程可以对信号产生响应就没有任何理由来唤醒所有的线程。

所以一定要记住，不要在全局对象，String常量上使用wait/notify/notifyAll方法。监视器(即被加锁的对象)必须对于它所属的对象是独立的。就像上面的MyWaitNotify3类一样，每个该类的对象都持有一个自己的MonitorObject对象，而不会在空的String常量上调用wait或notify方法。
